M3 LESSON

RAG（社内知識で正しさを作る）

LLMの知識の限界を補い、社内文書・FAQを根拠にした信頼性の高い回答を実現する

🔍 RAGとは何か ─ なぜ必要か

RAG（Retrieval-Augmented Generation / 検索拡張生成）は、LLMの「知識の限界」を補うための設計パターンです。LLMは学習データの範囲内でしか回答できず、社内固有の情報や最新情報は持っていません。RAGはこの問題を解決します。

LLMの「知識の限界」とは

  • カットオフ日： 学習データには期限があり、それ以降の出来事を知らない
  • 社内情報の欠如： Axxx Japan固有の規程・製品・組織情報は学習されていない
  • ハルシネーション： 知らないことに対して、もっともらしい誤答を生成してしまう

RAGの処理フロー

RAGは「質問→検索→コンテキスト注入→回答生成」の4ステップで動作します。

# RAGの基本フロー（概念コード）
# 1. ユーザーの質問を受け取る
user_question = "Axxx Japanのテレワーク規程はどうなっていますか？"

# 2. 質問を検索クエリに変換してベクターストアを検索
relevant_chunks = vector_store.search(user_question, top_k=3)

# 3. 検索結果をコンテキストとしてプロンプトに注入
context = "\n\n".join([chunk.text for chunk in relevant_chunks])
prompt = f"""
以下の社内文書を参照して質問に答えてください。
文書にない情報は「文書に記載がありません」と答えてください。

## 参照文書
{context}

## 質問
{user_question}
"""

# 4. LLMが参照文書に基づいて回答生成
answer = llm.generate(prompt)

RAGなし vs RAGありの回答品質比較

質問 RAGなし（LLM単体） RAGあり
「テレワーク手当はいくらですか？」 「一般的には月5,000〜10,000円程度...」（ハルシネーション） 「規程第15条に基づき月8,000円です（出典: 就業規則2026年版）」
「システムAの障害報告手順は？」 「一般的には、まずインシデント管理システムに...」（社内手順と異なる） 「SystemAops手順書v3に従い、まずSlack #incident に報告...」
RAGが「社内知識で正しさを作る」理由：
RAGはLLMに「根拠となる文書」を与えて回答させます。LLMは与えられた文書の範囲内で回答するよう誘導されるため、社内固有情報に基づいた回答が得られ、引用元を明示することでハルシネーションを大幅に低減できます。

📐 ベクトル検索の基本

RAGの「検索」部分を支えるのがベクトル検索です。キーワードの一致ではなく「意味的な近さ」でテキストを検索します。

テキストのベクトル化（Embeddings）

Embedding とは、テキストを数値のベクトル（高次元の数値列）に変換する技術です。意味が近いテキストほど、ベクトル空間で近い位置に配置されます。

# Embedding の取得（OpenAI Embeddings API）
from openai import AzureOpenAI
import numpy as np

client = AzureOpenAI(...)

# テキストをベクトルに変換
response = client.embeddings.create(
    model="text-embedding-3-small",
    input=["有給休暇の申請方法を教えてください"]
)
embedding = response.data[0].embedding  # 1536次元のベクトル
print(f"ベクトルの次元数: {len(embedding)}")    # → 1536

コサイン類似度による意味的な近さの計算

2つのベクトルの「向き」の近さを計算することで、テキスト同士の意味的な類似度を数値化します。

# コサイン類似度の計算
def cosine_similarity(vec_a: list[float], vec_b: list[float]) -> float:
    a = np.array(vec_a)
    b = np.array(vec_b)
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))

# 類似度の範囲: -1（完全反対）〜 0（無関係）〜 1（同じ意味）
query = "有給休暇の取得方法"
documents = [
    "年次有給休暇は人事システムから申請できます",  # → 高い類似度
    "サーバーのメンテナンス手順について",          # → 低い類似度
]

チャンキング戦略

文書を検索可能な単位（チャンク）に分割する際の戦略は、RAGの品質に大きく影響します。

チャンキング戦略 説明 向いている文書
固定サイズ 500トークンごとに分割（重複あり） 連続した説明文・マニュアル
文境界 句点・段落で分割 FAQ・プレスリリース
セクション単位 見出し（H1/H2）ごとに分割 規程書・仕様書・技術文書
チャンクサイズの目安：
  • 小さすぎる（50トークン以下）: コンテキストが失われ、検索精度が低下する
  • 大きすぎる（2,000トークン以上）: 関連部分が薄まり、LLMのコンテキスト窓を圧迫する
  • 推奨: 200〜500トークン（重複50トークンを持たせるとコンテキスト切れを防げる）

🏗️ RAGの実装パターン

RAGには複数の実装パターンがあります。ユースケースや要求品質に応じて選択します。

密ベクトル検索（Dense Retrieval）

最も基本的なパターンです。文書をすべてベクトル化しておき、クエリのベクトルと最も近い文書を返します。

# 密ベクトル検索の基本実装
class SimpleVectorStore:
    def __init__(self):
        self.chunks = []       # テキストチャンク
        self.embeddings = []   # 対応するベクトル

    def add(self, text: str, embedding: list[float]):
        self.chunks.append(text)
        self.embeddings.append(np.array(embedding))

    def search(self, query_embedding: list[float], top_k: int = 3):
        q = np.array(query_embedding)
        scores = [
            np.dot(q, e) / (np.linalg.norm(q) * np.linalg.norm(e))
            for e in self.embeddings
        ]
        top_indices = np.argsort(scores)[::-1][:top_k]
        return [(self.chunks[i], scores[i]) for i in top_indices]

再ランキング（Re-ranking）

ベクトル検索で候補を絞り込んだ後、Cross-encoderで再評価して精度を上げる手法です。

手法 速度 精度 用途
密ベクトル検索のみ 速い 中 リアルタイム検索・大規模文書
検索 + 再ランキング やや遅い 高 高精度が求められる用途

⚖️ RAGで解ける問題 / 解けない問題

RAGは万能ではありません。どのような問題に適しているかを正確に理解することが重要です。

RAGで解ける問題
社内FAQ・マニュアルへの質問 「有給休暇の申請方法は？」「経費精算の上限は？」
仕様書・設計書の内容確認 「システムAのAPIエラーコード一覧は？」
社内規程・ポリシーの照会 「情報セキュリティポリシー第3条の内容は？」
RAGで解けない問題
リアルタイム情報 「今日の株価は？」「現在のシステム障害状況は？」（DB直接参照が必要）
複雑な多段推論 「A部門とB部門の予算を比較して最適化案を提案して」（複数文書の統合・計算が必要）
知識ベースにない情報 文書化されていない暗黙知・口頭で共有された情報
RAGの適用判断フロー：
① 回答の根拠となる文書が存在するか？ → 存在しない場合はRAGではなく別の手段を検討
② 文書がデジタル化・テキスト化されているか？ → PDFや画像のみの場合はOCR・前処理が必要
③ 文書の更新頻度は？ → 頻繁に更新される場合は再インデックスの仕組みが必要

⚠️ よくある失敗パターン

RAGシステムの構築で頻繁に遭遇する失敗パターンと対策を理解しましょう。

失敗1: チャンク設計ミス

  • 大きすぎる： 1チャンク5,000文字以上 → 関連情報が薄まり、LLMのコンテキストを圧迫
  • 小さすぎる： 1チャンク50文字以下 → コンテキストが失われ、意味のある検索結果が得られない
  • コンテキストが切れる： 「これは〜」という代名詞でチャンクが始まる → 参照先が不明になる
対策： チャンクに前後の文脈（重複部分）を持たせる。見出しや文書名を各チャンクの先頭に付加して、単体で意味が通るようにする。

失敗2: 更新忘れ

ベクターストアに一度登録した文書を更新しないと、古い情報を参照し続けます。

対策： 文書更新時に自動でベクターストアを再インデックスするパイプラインを設ける。文書に「最終更新日」を含め、RAGの回答にも更新日を表示する。

失敗3: 引用がない回答

「どの文書から来たかわからない」回答は、検証が困難でハルシネーションのリスクがあります。

失敗4: 権限制御の欠如

全社員が同一のベクターストアにアクセスできると、他部門の機密文書（人事情報・M&A関連資料など）が混入するリスクがあります。

対策： ユーザーのロール・部門に応じてアクセス可能な文書を絞り込む「フィルタリング」機能を実装する。ベクターストアを部門ごとに分離することも有効。
失敗パターン 症状 対策
チャンク設計ミス 検索精度が低い・意味不明な回答 200〜500トークン・重複あり・見出し付加
更新忘れ 古い情報を堂々と提示する 自動再インデックスパイプライン
引用なし 根拠が確認できない プロンプトで引用を必須にする
権限制御の欠如 機密情報が誰でも参照できる ロールベースのフィルタリング
