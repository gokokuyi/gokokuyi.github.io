M0 LESSON

生成AIの前提（全員）

LLMの仕組み・限界・正しさの作り方・業務活用の基本姿勢を理解する

🧠 LLMとは何か ─ 確率的生成の仕組み

LLM（大規模言語モデル）は、インターネット上の大量のテキストデータを学習し、「次のトークン（単語・文字のかたまり）として何が来る確率が高いか」を予測することで文章を生成します。

トークン予測の仕組み

LLMは入力テキストを受け取ると、語彙全体（数万〜数十万トークン）に対して確率分布を計算し、その分布からサンプリングして次のトークンを選びます。これを終端トークン（</s>）が選ばれるまで繰り返します。

入力（プロンプト） 次のトークン候補（確率の例）
「東京の天気は」 晴れ: 40%、曇り: 25%、雨: 20%、不明: 15%
「東京の天気は晴れ」 です: 55%、でした: 20%、の: 15%、だ: 10%

同じ入力でも毎回出力が変わる理由

LLMが確率分布からサンプリングするため、同じプロンプトを与えても毎回異なる出力が生成されます。これを制御するのが temperature（温度） パラメータです。

  • temperature = 0: 常に最高確率のトークンを選ぶ（決定論的・再現性あり）
  • temperature = 0.7〜1.0: ランダム性が増し、創造的な表現が出やすい
  • temperature > 1.0: 低確率トークンも選ばれやすくなり、荒唐無稽な出力になりやすい
重要な視点： LLMは「考えている」わけではありません。膨大なテキストから学習した統計的パターンに基づき、「文脈に合いそうなトークン列」を生成しているに過ぎません。これがLLMの能力と限界の両方を説明する根本的な事実です。

「考えている」わけではない ─ 統計的パターンマッチ

LLMは以下のような「理解」や「推論」を本質的には行っていません：

  • 概念の意味を「わかって」いるわけではない（記号操作に近い）
  • 事実を「知って」いるわけではない（学習データに出現したパターンを記憶している）
  • 計算を「実行して」いるわけではない（計算式の見た目のパターンから回答を推測している）
たとえば「7839 × 4291 = ?」という問いに対して、LLMは実際に掛け算を実行せず、学習データ中の類似した掛け算の結果パターンから推測します。そのため計算を間違えることが多くあります。

⚖️ LLMの得意なこと / 不得意なこと

LLMを業務に活用するには、何が得意で何が苦手かを正確に把握することが不可欠です。

得意なこと

得意タスク 説明 例
文章生成・ドラフト 指示に沿った文章を高速に生成する メール下書き、提案書のたたき台
要約 長文を要点に絞って短くまとめる 議事録要約、インシデントレポート
翻訳 多言語間の翻訳（文脈を保ちながら） 技術文書の日英翻訳
コード補完・生成 仕様からコードのたたき台を生成する ボイラープレート生成、リファクタ提案
パターン認識・分類 テキストのカテゴリ分け・感情分析 問い合わせ種別の自動振り分け

不得意なこと（リスクが高い領域）

不得意タスク 理由 対策
最新情報の提供 学習データにはカットオフ日があり、それ以降の情報を知らない RAG（検索拡張生成）で最新文書を注入する
正確な計算 計算はパターンマッチで行うため誤りが生じる コードインタープリタ（ツール）に計算を委ねる
事実の根拠確認 「もっともらしい文章」を生成するため、誤った事実でも自信満々に述べる（幻覚：ハルシネーション） 引用元URLや原文との照合を必須にする
「わからない」と言えない 不確かな情報でも回答を生成してしまう傾向がある プロンプトで「わからない場合はわからないと答えよ」と明示する
ハルシネーション（幻覚）とは： 存在しない書籍のタイトル・著者名、架空のURLや法令番号、実際には発言していない人物の言葉などを、自信満々に「事実」として生成する現象です。Axxx Japanの社内業務においても、LLMが内部規程や手続きについて誤った情報を正しいかのように提示するリスクがあります。

🔍 「正しさ」をどう作るか ─ 参照（Retrieval）と評価（Eval）が本体

LLMは単体では「正しさ」を保証できません。本番で信頼できる出力を得るには、参照（Retrieval） と 評価（Evals） の2つの仕組みが必須です。

参照（RAG: Retrieval-Augmented Generation）

RAGは「外部の正しい知識をプロンプトに注入してから回答させる」設計パターンです。

# RAGの基本フロー（概念）
# 1. ユーザーの質問を受け取る
user_question = "有給休暇の申請方法は？"

# 2. 社内文書データベースから関連文書を検索する
relevant_docs = search_internal_docs(user_question)

# 3. 検索結果をプロンプトに埋め込む
prompt = f"""
以下の社内文書を参照して質問に答えてください。

## 参照文書
{relevant_docs}

## 質問
{user_question}
"""

# 4. LLMに回答させる（根拠のある回答が得られる）
answer = llm.generate(prompt)
RAGの効果：
  • 学習カットオフ以降の情報（最新規程・社内情報）を参照できる
  • 回答に「どの文書から引用したか」を明示できる
  • ハルシネーションを大幅に低減できる

評価（Evals: Evaluations）

Evalsは「LLMの出力品質を継続的にテストし続ける仕組み」です。ソフトウェアのユニットテストに相当します。

Eval の種類 説明 例
自動Eval（ルールベース） 正規表現・完全一致・包含チェックで自動採点 「回答に電話番号が含まれているか」
自動Eval（LLMベース） 別のLLMが回答の正確さ・適切さを採点 「回答は社内規程に沿っているか（5点満点）」
手動Eval 人間がサンプルを目視確認して品質を保証 週次でランダム50件をレビューする
なぜEvalsが「本体」なのか：
プロンプトを変更した際に「品質が上がったか下がったか」を判断するには、定量的なEvalsが必要です。Evalsなしでプロンプトを改修すると、改善なのか劣化なのかが判断できなくなります。RAGとEvalsの2つがないまま本番稼働させると、品質劣化に気づかずユーザーに誤情報を提供し続けるリスクがあります。

🏢 AIを業務に使うときの基本姿勢

LLMの仕組みと限界を踏まえ、業務に導入する際に守るべき基本原則を理解しましょう。

原則1: LLMの回答を一次情報として使わない

LLMが生成した情報は、学習データのパターンから生成されたものであり、事実の正確性は保証されていません。以下のことを徹底してください。

  • LLMの回答を、原典・公式ドキュメント・社内規程などと必ず照合する
  • 「LLMがそう言っていた」という理由だけで意思決定しない
  • 外部への回答・報告書への引用には、原典を確認してから使用する
Axxx Japanでの具体例：
社内規程についての質問をLLMにして、回答をそのままメールに書いてはいけません。必ずイントラネットの規程ページで確認してから使用してください。

原則2: 人間がレビューする工程を必ず残す

LLMを使った自動化の設計において、以下のフローを守ってください：

  • ドラフト生成はLLMに任せる ─ 初稿・たたき台・候補リストの生成
  • 最終判断・送付・確定は人間が行う ─ 担当者が内容を確認してから実行
  • 高リスク領域は二重確認 ─ 顧客向け回答・法的文書・金額に関わるものは複数人でレビュー

原則3: 用途・リスク・責任範囲を明確にしてから始める

AI導入前に以下を明文化する習慣をつけましょう：

確認項目 確認内容
用途 何の業務に、誰が、どのように使うか
リスク 誤答したとき何が起きるか（顧客影響・金融影響・法的影響）
責任範囲 誤りが生じた場合、誰が責任を持つか
情報の機密性 プロンプトに入力する情報は社外に出て良いか（クラウドLLMの場合）
品質基準 どの程度の精度・品質があれば「使える」と判断するか
まとめ：AI活用の3原則
① LLMの回答は必ず原典照合する
② 最終判断は人間が行う
③ 用途・リスク・責任を明文化してから始める
