M6 LESSON

評価（Evals）が品質保証

LLMアプリを「なんとなく動かす」から「測定して改善する」へ

📊 なぜ評価（Evals）が重要か

LLM導入の失敗のほとんどは「評価がない」から始まる

社内でAIアシスタントを導入した直後は「すごい！」と感動します。 しかし数ヶ月後に「なんか精度が落ちた気がする」「あのプロンプト変更から回答の質が変わった」 と感じても、評価の仕組みがなければ何が原因かわかりません。

多くのLLM導入プロジェクトが陥る罠があります。

  • 担当者が試して「よさそう」と感じたら本番投入する
  • プロンプトを変更したが、改善したかどうかを測定しない
  • ユーザーから「おかしい」とクレームが来て初めて問題に気づく
  • 使っているモデルのバージョンが変わって品質が変化したことに気づかない
なぜ評価がないと危険か： LLMの出力は確率的であり、同じ入力でも毎回少し違う出力が返ります。 また、モデルのバージョン更新・プロンプトの変更・RAGのデータ更新など、 品質に影響する変数が多数あります。 評価がなければ、これらの変化がプラスに働いているのかマイナスに働いているのかわかりません。

「なんとなく動いている」から「測定して改善する」へ

ソフトウェア開発では「テストなしの本番リリース」は考えられません。 LLMアプリでも同じです。評価（Evals）は LLMアプリのテスト です。

評価を導入することで得られるメリット：
① プロンプト変更前後の品質を客観的に比較できる
② モデルのバージョン更新による品質変化を検知できる
③ 回答精度の数値目標を設定・追跡できる
④ 問題のある回答パターンを体系的に発見できる

プロンプト変更・モデル変更・知識更新で品質が劣化するリスク

LLMアプリの品質は静的ではありません。以下のような変更がトリガーとなって品質が劣化します。

変更の種類 品質劣化の具体例
プロンプト変更 ある質問カテゴリの精度が上がったが、別のカテゴリが大幅に悪化した
モデルバージョン更新 新しいモデルは英語精度が高いが、日本語の特定ドメインで精度が低下した
RAGデータ更新 FAQ のリライトにより、検索でヒットするチャンクの内容が変わり回答精度が変化した

🔬 評価手法の体系

LLMアプリの評価手法は、速度・コスト・信頼性のトレードオフによって複数の手法があります。 用途に応じて組み合わせて使うことが重要です。

① ルールベース評価

正規表現マッチ・キーワード含有確認・JSON構造バリデーションなど、 明確な正解基準をコードで記述できる場合に使います。 

# キーワード含有チェックの例
def check_keywords(response: str, expected: list[str]) -> bool:
    return all(keyword in response for keyword in expected)

# JSON スキーマ検証の例
import json
def validate_json(response: str, required_keys: list[str]) -> bool:
    try:
        data = json.loads(response)
        return all(key in data for key in required_keys)
    except json.JSONDecodeError:
        return False

② 類似度評価

参照回答（ゴールデン回答）と生成された回答の類似度を計算します。

  • ROUGE：n-gram の共起率を評価（要約の評価によく使われる）
  • BLEU：機械翻訳の評価指標を転用（精度重視）
  • コサイン類似度：埋め込みベクトルの類似度（意味的な近さを評価）
ROUGE/BLEU は語彙レベルの一致を測定するため、 意味が同じでも表現が違うと低スコアになる弱点があります。 埋め込みモデルを使ったコサイン類似度は意味的な近さを評価できますが、 計算コストがかかります。

③ LLM-as-judge（LLMによる採点）

評価専用の LLM プロンプトを設計し、別の LLM が回答品質を採点します。 人間の評価者に近い柔軟性があり、自然言語の品質評価に優れています。

④ 人手評価（ゴールデンデータセット）

専門家が作成した「ゴールデンデータセット」（正解回答付きのテストケース集）と モデルの回答を比較します。最も信頼性が高いですが、作成・維持コストが高いです。

各手法のトレードオフ

手法 速度 コスト 信頼性 適用場面
ルールベース ⚡ 高速 💰 低 明確な正解がある場合は高い JSON構造・キーワード確認
類似度評価 ⚡ 高速 💰 低〜中 中程度（表現揺れに弱い） 要約・翻訳の品質評価
LLM-as-judge 🐌 中程度 💰💰 中〜高 高い（バイアス注意） 自由形式の回答品質評価
人手評価 🐌 低速 💰💰💰 高 最高 最終確認・重要指標の校正

⚖️ LLM-as-judge の設計

評価プロンプトの書き方

LLM-as-judge では、評価基準・スケール・Chain-of-Thought（COT） の3要素を評価プロンプトに含めることが重要です。 

JUDGE_PROMPT = """
# タスク
以下のAI回答の品質を評価してください。

# 評価基準
- 正確性: 情報が正確か（1-5）
- 簡潔性: 不要な情報がないか（1-5）
- 実用性: ユーザーの問題を解決できるか（1-5）

# 質問
{question}

# AI回答
{answer}

# 出力形式（必ずこのJSON形式で返してください）
{{
  "scores": {{"accuracy": <1-5>, "conciseness": <1-5>, "usefulness": <1-5>}},
  "overall": <1-5>,
  "reason": "<評価の根拠を1-2文で>"
}}
"""

3つの主要バイアスへの注意

PositionBias（位置バイアス）： 複数の回答を比較するとき、先に提示された回答を高評価する傾向。 → 対策：評価順をランダム化する
LengthBias（長さバイアス）： 長い回答の方が詳しく見えて高評価される傾向。 → 対策：簡潔性を評価基準に明示し、COTで根拠を説明させる
SelfEnhancementBias（自己強化バイアス）： 評価者と同じモデルが生成した回答を高評価する傾向。 → 対策：異なるモデル・バージョンを評価者に使う

参照回答ありの評価（G-eval）vs 参照回答なし（pairwise）

手法 概要 適用場面
G-eval（参照回答あり） 正解回答と比較しながら採点する ゴールデンデータセットがある場合
pairwise（参照回答なし） 2つの回答を比較してどちらが良いか判定する A/Bテスト・プロンプト改善の比較

スコアリング設計の原則

評価スコアは 1〜5 点スケール が扱いやすく、 各スコアの意味を定義しておくことで評価の一貫性が高まります。 

# スコア定義の例
SCORE_RUBRIC = """
5: 完全に正確で簡潔、ユーザーの問題を完全に解決する
4: ほぼ正確で実用的、軽微な不備がある
3: 部分的に正確、改善の余地がある
2: 不正確または不完全で実用性が低い
1: 完全に不正確または有害な情報を含む
"""

🔄 回帰テストの設計

モデル・プロンプト変更のたびに走らせるテストスイート

ソフトウェア開発の 回帰テスト（変更が既存機能を壊していないか確認するテスト）を LLMアプリにも適用します。 プロンプトやモデルを変更するたびに、決まったテストケース集を実行して品質が維持されているか確認します。 

def run_regression_test(client, model: str, test_cases: list[dict]) -> dict:
    results = {"total": len(test_cases), "passed": 0, "failed": 0}
    for case in test_cases:
        response = client.responses.create(model=model, input=case["question"])
        answer = response.output_text
        # 期待するキーワードが回答に含まれるか確認する
        passed = all(kw in answer for kw in case["expected_keywords"])
        results["passed" if passed else "failed"] += 1
    return results

評価セットの作り方

良い評価セットを作るには、実際のユーザーの質問・問題のあった回答・FAQデータから収集するのが最も効率的です。

  • 実チケット収集：カスタマーサポートのQ&Aから実際の質問を収集する
  • FAQからの変換：既存のFAQドキュメントを問い形式に変換してテストケース化する
  • エッジケースの追加：本番で問題になったケースを随時追加する
「人工的に作ったテストケース」よりも「実際にユーザーが質問した内容」のほうが、 本番環境での品質を正確に反映します。 まずは20〜50件の実質問から始め、徐々に拡充していきましょう。

CI/CDパイプラインへの組み込み方針

プロンプトやシステムプロンプトをバージョン管理（Git）している場合、 プルリクエスト時に自動で評価テストを実行する仕組みを作ることができます。 

# CI/CD の評価ステップ例（GitHub Actions など）
# 1. プロンプトの変更を検知する
# 2. 回帰テストを自動実行する
# 3. 合格率が閾値（例: 85%）を下回ったらマージをブロックする
# 4. 結果をプルリクエストにコメントする

♻️ 継続的評価サイクル

ゴールデンデータセットの育て方

ゴールデンデータセットは一度作って終わりではなく、継続的に育て続けるものです。 最も効果的な方法は誤答をケースに追加することです。

  • ユーザーから「この回答は間違っている」というフィードバックがあったケースを追加する
  • 本番ログを定期的にレビューして、品質が低い回答を発見してケース化する
  • 新しいFAQや社内ドキュメントを追加したときは対応するテストケースも追加する

メトリクス化する指標

指標 意味 目標値の例
精度（Accuracy） 評価セットに対する合格率 85%以上を維持する
カバレッジ 回答できた割合（「わかりません」以外） 90%以上
応答時間 ユーザーへの回答までの時間 P95 で 5秒以内
コスト/クエリ 1回の回答にかかるAPIコスト 月次予算内に収める

人手評価のサンプリング戦略

全ての本番クエリを人手評価するのはコストが高すぎます。 以下のようなサンプリング戦略で効率的に品質を監視します。

  • ランダムサンプリング：全クエリの5%を毎週ランダム抽出して人手評価する
  • 低スコアサンプリング：LLM-judgeが低スコアをつけた回答を優先的に人手評価する
  • ユーザーフィードバック起点：「役に立たなかった」のフィードバックがあったケースは必ず確認する
継続的評価サイクルのまとめ：
本番ログ収集 → LLM-judge による自動評価 → 低品質ケースを人手確認 → 問題ケースをゴールデンデータセットに追加 → 回帰テストを実行 → プロンプト・データ改善 → 繰り返す
