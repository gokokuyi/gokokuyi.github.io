# AIエンジニア育成教材「AI活用人材育成カリキュラム」のご紹介

> 対象読者：社内DX推進担当・研修企画担当・AI活用に取り組むエンジニアの方々

---

## はじめに

「ChatGPTを使って業務効率化したい」——そう思って試してみたものの、  
**「どこまで任せていいの？」「精度が悪いのはなぜ？」「社内システムと繋ぐにはどうする？」**  
といった壁にぶつかった経験はないでしょうか。

APIを叩けることと、**実務で使えるAIアプリを設計・実装・評価できること**は、まったく別のスキルです。

今回ご紹介する「**AI活用人材育成カリキュラム**」は、その差を埋めるために設計された社内研修向けのWebベース教材です。

---

## こんな方に向けた教材です

- Pythonの基本文法は書けるが、LLMアプリ開発の経験はない
- ChatGPTは使っているが、APIの仕組みを理解していない
- RAGやFunction Callingという言葉を聞いたことはあるが、実装したことはない
- 社内にAI活用を広めるための研修コンテンツを探している

---

## カリキュラムの全体像：M0 〜 M6 の7モジュール

教材はM0（前提知識）からM6（評価・品質保証）まで、段階的に積み上がる構成になっています。

| Module | テーマ | 内容 | 難易度 |
|--------|--------|------|--------|
| **M0** | 生成AIの前提 | LLMの仕組み・限界・ハルシネーション・業務活用の心構え | ★☆☆ |
| **M1** | ユースケース選定 | AIに向く仕事・向かない仕事の見極め方、効果試算 | ★☆☆ |
| **M2** | プロンプト設計 | 出力制御・システムプロンプト・Few-shot・Structured Outputs | ★★☆ |
| **M3** | RAG | Embeddings・コサイン類似度・ベクトル検索・社内知識の活用 | ★★★ |
| **M4** | ツール利用 | Function Calling・MCP・業務システム連携・エージェント設計 | ★★★ |
| **M5** | セキュリティ | PII検出・プロンプトインジェクション・OWASP LLM Top 10 | ★★★ |
| **M6** | 評価（Evals） | LLM-as-Judge・自動テスト・回帰テスト・品質保証のしくみ | ★★★ |

### 最終成果物

全モジュールを修了すると、以下のような社内向けAIアシスタントを一から実装できるスキルが身につきます。

```
社内 Q&A ボット（MCP サーバ対応版）
├── CLIから自然言語で質問できる              ← M2
├── 社内FAQをRAGで検索して根拠ある回答を生成   ← M3
├── 検索が必要かどうかをAIが自律判断する       ← M4
├── PII（個人情報）を安全に処理する            ← M5
└── 回答品質を自動テストで継続的に計測する      ← M6
```

## 教材の学習スタイル：Lesson → Quiz → Handson

各モジュールは3ステップで構成されています。

### 📖 Lesson（知識インプット）
- LLMの仕組み・設計の考え方を図解・比喩を交えながら解説
- 「なぜそう設計するのか（Why）」を必ず説明
- サンプルコード（`.sample.py`）で実装の全体像を先に把握

### ❓ Quiz（理解確認）
- 各モジュール10問の確認テスト
- 選択式・穴埋め式・考察問題を組み合わせた構成
- 「1問ずつ答えを確認」または「全問一括表示」に対応

### 💻 Handson（実装練習）
- `# EDIT HERE!` のコメントが付いた**穴埋め形式のPythonコード**
- 3段階ヒント（① APIの名前 → ② 引数の使い方 → ③ 完全な解答例）
- 実際にAzure OpenAI Service（またはOpenAI API）に接続して動作確認
- 完了チェックリストで自己評価

## 実装で触れる技術スタック

```
Azure OpenAI Service / OpenAI API
├── Responses API（テキスト生成・会話履歴管理）
├── Embeddings API（ベクトル化・類似度計算）
├── Function Calling / Tool Use（ツール定義・自律判断）
├── Structured Outputs（JSON形式の出力保証）
└── MCP（Model Context Protocol）サーバ実装（FastMCP）
```

## ピックアップ：各モジュールの見どころ

### M0「生成AIの前提」— 全員が最初に学ぶべきこと

LLMが「確率的にトークンを生成する仕組み」から始まり、ハルシネーションの発生原因・知識カットオフ・業務でのリスク管理まで体系的に学びます。  
「AIは全部知っている」「AIが言うから正しい」という誤解を解くところから始めるため、開発者でない方にも対象を広げています。

### M3「RAG」— 社内知識で正しさを作る

LLMは社内固有の情報を知りません。RAG（検索拡張生成）はこの限界を解決する設計パターンです。  
このモジュールでは**コサイン類似度の数式から実装まで**を丁寧に解説し、社内FAQをベクトル化して検索するチャットボットを実際に構築します。

### M4「ツール利用」— AIに仕事をさせる

Function CallingでAIが「検索すべき質問かどうか」を自律的に判断する仕組みを実装します。  
さらに**MCP（Model Context Protocol）**を使ったサーバ実装まで踏み込み、複数のAIツールから呼び出せる共通インターフェースの設計を学びます。

### M5「セキュリティ」— 安全なAIアプリの設計

個人情報（PII）の検出・マスキング、プロンプトインジェクション攻撃への対処、OWASP LLM Top 10に基づくリスク管理を実装を通じて習得します。  
「動くAIアプリ」から「本番投入できる安全なAIアプリ」への視点転換を促すモジュールです。

### M6「評価（Evals）」— なんとなく動かすから卒業する

LLMアプリの品質は「試して良さそうなら本番投入」では担保できません。  
**LLM-as-Judge**（LLMがLLMの出力を採点する手法）・自動回帰テスト・スコア可視化を実装し、プロンプト変更やモデル更新による品質変化を定量的に検知できる仕組みを構築します。

## 教材設計のこだわり

### 「Why」を欠かさない
コードの書き方だけでなく、**なぜその設計をするのか**を必ず説明しています。  
例：「RAGのチャンクサイズをなぜ2048トークンにするのか」「Function CallingのdescriptionにLLMが呼ぶかどうかを判断させるためなぜ具体的な説明が必要なのか」

### 架空のシナリオで一貫したストーリー
「Axxx Japan IT推進部に配属された新入社員・田中が社内Q&Aボットを段階的に構築する」  
という架空のシナリオが全モジュールを貫いており、学習が断片的にならないよう設計されています。

### Webブラウザだけで受講可能
HTMLベースの静的サイトのため、**環境構築なしにLessonとQuizを受講**できます。  
Handsonのみ Python + Azure OpenAI の実行環境が必要です。

## 受講を始めるには

1. `src/AI-learning/index.html` をブラウザで開く
2. M0 から順に **Lesson → Quiz → Handson** の順で進める
3. Handsonを実施する場合は `src/AI-learning/handson/` 内の演習ファイルを使用

```bash
# 実行環境のセットアップ（初回のみ）
cd src/AI-learning/handson
pip install -r requirements.txt
cp config.yaml.example config.yaml
# config.yaml に Azure OpenAI の接続情報を設定してください
```
