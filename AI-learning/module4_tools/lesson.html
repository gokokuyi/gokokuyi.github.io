M4 LESSON

ツール利用（業務システムと繋ぐ）

Function Calling と MCP でAIを業務システムに接続し、安全なエージェントを設計する

🔌 ツール利用（Function Calling）の基本

ツール利用（Function Calling）とは、LLMが自律的に「どの関数を、どの引数で呼ぶか」を判断し、外部の処理を実行させる仕組みです。「AIに何かをやらせる」ためのインターフェースです。

ツール定義の構造

ツールは name（名前）・description（説明）・parameters（引数の定義）の3要素で定義します。LLMはこの定義を読んで「このツールをいつ使うか」を判断します。

# ツール定義の例（チケット検索ツール）
tool_search_ticket = {
    "type": "function",
    "function": {
        "name": "search_ticket",
        "description": "社内チケットシステムでチケットを検索します。キーワードまたはステータスで絞り込めます。",
        "parameters": {
            "type": "object",
            "properties": {
                "keyword": {
                    "type": "string",
                    "description": "検索キーワード"
                },
                "status": {
                    "type": "string",
                    "enum": ["open", "in_progress", "closed"],
                    "description": "チケットのステータス"
                }
            },
            "required": ["keyword"]
        }
    }
}

AIはいつツールを使うかを自律判断する

ツール定義を渡すだけで、LLMはユーザーのメッセージを読み「このツールを呼ぶべきか、呼ぶなら何の引数を使うか」を自律的に判断します。

ユーザーの入力 LLMの判断
「ログイン不可のチケットを探して」 search_ticket(keyword="ログイン不可") を呼び出す
「オープン中のチケット一覧を見せて」 search_ticket(keyword="", status="open") を呼び出す
「AIについて教えて」 ツールを使わず直接回答する
ツール利用の3ステップ：
① LLM が tool_calls を返す（関数名と引数）
② アプリケーション（コード）が実際の関数を実行する
③ 実行結果を role: "tool" メッセージとして LLM に渡し、最終回答を得る

🛠️ ツールとして抽象化できるもの

業務システムの様々な操作を「ツール」として抽象化することで、LLMがこれらを呼び出して業務タスクを自律的に処理できるようになります。

カテゴリ ツール例 業務シナリオ
チケットシステム create_ticket, search_ticket, update_ticket 障害報告の自動チケット起票
データベース検索 search_faq, lookup_customer FAQへの自動回答、顧客情報照会
社内API submit_expense, book_room 経費申請、会議室予約
計算・変換処理 calculate_tax, convert_currency 税計算、通貨換算

ツール定義で重要な「description」

LLMはツールの description を読んで使用判断をします。説明が曖昧だと、ツールを使うべき場面で使わなかったり、使うべきでない場面で使ったりします。

# 悪い例: description が曖昧
{"name": "get_data", "description": "データを取得します"}

# 良い例: 使用条件・引数・戻り値が明確
{"name": "get_expense_limit",
 "description": "従業員IDと経費カテゴリを受け取り、そのカテゴリの上限金額（円）を返します。
                   カテゴリ: 交通費、宿泊費、会議費、接待費"}

🔒 過剰権限のリスクと段階的権限付与

ツールにどの権限を与えるかは、セキュリティ設計の中で最も重要な判断の一つです。過剰な権限を持つエージェントは、誤動作時や悪意ある入力（プロンプトインジェクション）の際に取り返しのつかない損害を引き起こす可能性があります。

「読み取り専用から始める」原則

権限は段階的に付与します。書き込み・削除・外部送信は慎重に検討してから追加します。

フェーズ 許可する操作 判断基準
フェーズ1 読み取りのみ（search, get, list） まず始める。誤動作のリスクが低い
フェーズ2 作成（create）の追加 作成物は後から確認・修正できる
フェーズ3 更新・削除（update, delete）の追加 不可逆な操作。Human-in-the-loop を必ず実装する
フェーズ4 外部送信（email, notify）の追加 外部への影響。必ず人間の承認フローを経る

Human-in-the-loop（重要アクションに人間の確認を挟む）

削除・外部送信・金額の大きい申請など、取り消しが困難な操作の前には必ず人間の確認ステップを設けます。

# Human-in-the-loop の実装例
def delete_ticket(ticket_id: str) -> dict:
    # 削除前に人間の確認を求める
    ticket = get_ticket(ticket_id)
    print(f"\n⚠️ 以下のチケットを削除します:")
    print(f"  ID: {ticket['id']}, タイトル: {ticket['title']}")
    confirm = input("削除してよいですか？ (yes/no): ").strip().lower()

    if confirm != "yes":
        return {"status": "cancelled", "message": "ユーザーによりキャンセルされました"}

    # 承認後に実際の削除を実行
    return execute_delete(ticket_id)
OWASP LLM06: 過剰な権限（Excessive Agency）：
LLMエージェントに過剰な権限を与えることは、OWASPが定めるLLMアプリケーションのリスクの一つです。最小権限の原則（Principle of Least Privilege）に従い、タスクの実行に必要な最小限のツールと権限のみを付与してください。

🌐 MCP（Model Context Protocol）入門

MCP（Model Context Protocol）は、ツールをサーバーとして外部公開するための標準プロトコルです。2024年にAnthropicが提案し、OpenAIを含む主要プレイヤーが採用しています。

MCPとは何か

従来のFunction Callingでは、ツールの実装とLLMの呼び出しコードが密結合していました。MCPではツールをサーバーとして分離することで、再利用性・セキュリティ・保守性が大幅に向上します。

観点 従来の Function Calling MCP
構成 ツール定義とLLM呼び出しが同一コード内 ツールはサーバーとして分離・独立
再利用性 プロジェクトごとに再実装が必要 MCPサーバーを複数のAIから利用可能
セキュリティ 権限制御が実装次第 サーバー側で権限・認証を一元管理

FastMCP でサーバーを実装する

FastMCP は Python でMCPサーバーを簡単に実装できるライブラリです。

# FastMCP によるMCPサーバーの実装例
from fastmcp import FastMCP

mcp = FastMCP("Axxx Japan-社内ツールサーバー")

# デコレータでツールを定義
@mcp.tool()
def search_ticket(keyword: str, status: str = "open") -> dict:
    """チケットシステムで検索します。keyword: 検索キーワード、status: open/closed"""
    # 実際の処理（DBアクセス等）
    tickets = ticket_db.search(keyword=keyword, status=status)
    return {"tickets": tickets, "count": len(tickets)}

@mcp.tool()
def get_expense_limit(category: str) -> dict:
    """経費カテゴリの上限金額を返します。category: 交通費/宿泊費/会議費"""
    limits = {"交通費": 50000, "宿泊費": 15000, "会議費": 30000}
    return {"category": category, "limit": limits.get(category, 0)}

if __name__ == "__main__":
    mcp.run(transport="streamable-http", host="127.0.0.1", port=8000)

🛡️ 設計のまとめ ─ 安全なエージェント設計

ツール利用エージェントを本番で稼働させる際の設計原則をまとめます。

ツール数は最小限に

LLMに与えるツール数が多いほど、誤った選択をするリスクが増えます。「タスクに必要なツールのみを与える」ことが基本原則です。

エラーハンドリングとフォールバック

ツールが失敗したとき（タイムアウト・認証エラー・API障害）の処理を必ず実装します。

# エラーハンドリングの実装例
def safe_tool_call(tool_name: str, args: dict) -> dict:
    try:
        result = execute_tool(tool_name, args)
        return {"status": "success", "result": result}
    except TimeoutError:
        return {"status": "error", "message": "ツールがタイムアウトしました。後でやり直してください。"}
    except PermissionError:
        return {"status": "error", "message": "権限がありません。管理者に確認してください。"}
    except Exception as e:
        return {"status": "error", "message": f"予期しないエラー: {str(e)}"}

監査ログ ─ どのツールをいつ呼んだかを記録

エージェントが実行したツール呼び出しはすべてログに記録します。問題発生時の原因調査と、セキュリティ監査に不可欠です。

# 監査ログの実装例
import json
from datetime import datetime

def log_tool_call(user_id: str, tool_name: str, args: dict, result: dict):
    log_entry = {
        "timestamp": datetime.utcnow().isoformat(),
        "user_id": user_id,
        "tool_name": tool_name,
        "args": args,
        "result_status": result.get("status"),
    }
    with open("audit.log", "a") as f:
        f.write(json.dumps(log_entry, ensure_ascii=False) + "\n")
安全なエージェント設計の5原則：
  1. 最小権限： 必要最小限のツールと権限のみを付与する
  2. 読み取り優先： 書き込み・削除は慎重に追加する
  3. Human-in-the-loop： 不可逆な操作は必ず人間の確認を経る
  4. エラーハンドリング： ツール失敗時の処理を必ず実装する
  5. 監査ログ： すべてのツール呼び出しを記録する
