M2 LESSON

プロンプト設計（再現性の型）

「なんとなく動く」から「再現性のある型」へ ─ 品質・保守性を高める構造化プロンプト設計

✏️ プロンプト設計の重要性 ─ 「なんとなく動く」から「再現性のある型」へ

LLMは同じタスクを依頼しても、プロンプトの書き方によって出力品質が大きく変わります。「なんとなくうまくいった」プロンプトは再現性がなく、担当者が変わったり条件がわずかに変わったりすると品質が劣化します。

書き方で結果がどう変わるか

たとえば「会議の議事録を要約してほしい」というタスクを例に取ります。

プロンプトの種類 例文 問題点
曖昧なプロンプト 「議事録を要約して」 長さ・形式・対象読者が不明。毎回バラバラな出力になる
構造化プロンプト 「以下の議事録を200字以内で日本語箇条書き3点にまとめてください。対象読者は経営層です」 なし ─ 制約・形式・対象が明確

再現性・品質・保守性を高める3つの柱

  • 再現性： 同じプロンプトを実行すれば誰でも同じ品質の出力が得られる
  • 品質： 出力フォーマット・内容の要件を明示することで品質が安定する
  • 保守性： プロンプトを文書化・バージョン管理することで改善が追跡できる
なぜ「型」が必要なのか：
属人的な「コツ」に頼ったプロンプトは、担当者交代・モデル更新・入力データの変化のたびに品質が崩れます。 構造化された型を持つことで、品質劣化の原因特定と改善が格段に容易になります。

📋 入力/出力の仕様化

プロンプトを「仕様書」として扱うことが、再現性のある出力を得る第一歩です。特に出力フォーマットの固定と制約の明示が重要です。

フォーマット固定（JSON・箇条書き・表）

後続処理でLLMの出力を使う場合、フォーマットを明示的に指定することが不可欠です。

# 悪い例: フォーマット指定なし
prompt_bad = """
以下の商品レビューの感情を分析してください。
レビュー: 「配送が遅くて残念でしたが、商品自体は良かったです」
"""

# 良い例: JSON形式を指定
prompt_good = """
以下の商品レビューの感情を分析し、JSON形式で回答してください。

## 出力形式
{
  "sentiment": "positive" | "negative" | "mixed",
  "score": 1〜5の整数,
  "reason": "50字以内の理由"
}

## レビュー
「配送が遅くて残念でしたが、商品自体は良かったです」
"""
フォーマット固定のメリット：
  • 出力をそのまま json.loads() でパースして後続処理に渡せる
  • 欠損フィールドや型違いを早期に検出できる
  • プロンプトの意図が明確になり、ハルシネーションが減る

制約の明示（文字数・言語・範囲）

LLMはデフォルトでは「適度な長さ」で回答しますが、業務用途では制約の明示が必要です。

制約の種類 プロンプト例
文字数 「100字以内で」「3文以内で」「箇条書き5点で」
言語 「必ず日本語で」「英語のみで回答してください」
スコープ 「以下の文書の範囲内でのみ回答してください」
対象読者 「非エンジニア向けに専門用語を使わず」「経営層向けに」

例示（Few-shot）の効果

Few-shotとは「このような入力にはこのような出力を返してほしい」という例をプロンプトに含める手法です。ゼロショット（例なし）より出力品質が大幅に向上します。

# Few-shot プロンプトの例
prompt_fewshot = """
問い合わせ文を「技術」「契約」「一般」のいずれかに分類してください。

## 例
入力: 「APIの認証エラーが発生しています」
出力: 技術

入力: 「契約の更新手続きを教えてください」
出力: 契約

入力: 「担当者を変更したい」
出力: 一般

## 分類対象
入力: 「ダッシュボードのグラフが表示されません」
出力:
"""
Few-shotが効く理由： LLMは例から「このタスクで何が期待されているか」を学習します。 特にエッジケースや判断が難しいケースの例を含めることで、分類精度が大幅に向上します。

🎯 役割付与より「タスク・評価基準・禁止事項」を明確に

「あなたは〇〇のエキスパートです」という役割付与（ペルソナ指定）は広く使われていますが、実際には具体的なタスク定義の方が出力品質に大きく影響します。

タスクの具体的定義

役割より「何をすべきか」を明確にする方が効果的です。

アプローチ 例 評価
役割付与のみ 「あなたはベテランの法務エキスパートです」 △ 何をすべきか不明
タスク定義 「以下の契約書から違約金条項を抽出し、リスクを高・中・低で評価してください」 ◎ 具体的で再現性が高い

評価基準を含める ─ 自己チェックさせる

プロンプト内に「良い出力の条件」を記述することで、LLMが自己評価しながら出力を改善します。

# 評価基準を含むシステムプロンプト例
system_prompt = """
## タスク
ユーザーの技術的な質問に回答してください。

## 評価基準（良い回答の条件）
- 具体的なコード例を含む
- 500字以内に収める
- 「かもしれません」「思います」などの曖昧な表現を使わない
- 参照すべき公式ドキュメントのURLを含む

## 禁止事項
- 質問の範囲を超えた余計なアドバイスをしない
- 実際に存在しないURLを記載しない
- 英語で回答しない（必ず日本語で）
"""

禁止事項で誤動作を防ぐ

LLMはデフォルトでは「役に立とう」とするあまり、求めていない情報を付け加えたり、範囲を超えた回答をすることがあります。禁止事項を明示することでこれを防ぎます。

禁止事項の書き方のコツ：
  • 「〇〇しないでください」より「〇〇は禁止」の方が明確
  • 禁止事項は最後に配置すると効果が高い（LLMは直前の指示の影響を受けやすい）
  • 禁止すべき具体例を1つ書くと効果が増す

🔧 構造化出力（JSON等）で後段処理を安定化

LLMの出力をコードで処理する場合、JSON形式での出力が最も安全です。OpenAI APIにはこれを強制する機能があります。

なぜJSON形式が必要か

自然言語の出力をプログラムで処理しようとすると、正規表現やパース処理が複雑になり、エラーが頻発します。JSON形式を強制することでこれを回避できます。

# response_format で JSON を強制する（Chat Completions API）
import json
from openai import AzureOpenAI

response = client.chat.completions.create(
    model=MODEL,
    response_format={"type": "json_object"},
    messages=[
        {"role": "system", "content": "JSONで回答してください"},
        {"role": "user", "content": "東京の気候を {city, climate, avg_temp} で返してください"}
    ]
)

# 安全にパースできる
result = json.loads(response.choices[0].message.content)
print(result["city"])  # → "東京"

Pydantic モデルでスキーマを定義する

OpenAI Structured Outputs を使うと、Pydanticモデルで定義したスキーマに完全準拠した出力を得られます。フィールドの欠損や型違いが原理的に発生しません。

from pydantic import BaseModel
from openai import AzureOpenAI

# スキーマ定義
class ReviewAnalysis(BaseModel):
    sentiment: str        # "positive" / "negative" / "mixed"
    score: int            # 1〜5
    key_points: list[str] # 主要ポイントのリスト
    recommendation: str   # 改善提案

# Structured Outputs で型安全な出力
response = client.beta.chat.completions.parse(
    model=MODEL,
    messages=[{"role": "user", "content": "レビュー: '使いやすいが価格が高い'"}],
    response_format=ReviewAnalysis
)

result = response.choices[0].message.parsed
print(result.sentiment)    # 型安全にアクセスできる
print(result.score)
print(result.key_points)   # list[str] であることが保証される
Structured Outputs の利点：
① フィールド欠損ゼロ（スキーマに従わない出力は自動的に修正される）
② 型安全（int のはずが str になるような事故がない）
③ IDE の補完が効く（result.score のような型付きアクセス）

🤝 Anthropic vs OpenAI のプロンプト設計思想

主要なLLMプロバイダーはそれぞれ異なるプロンプト設計の思想を持っています。どちらを使う場合でも共通の原則があります。

Anthropic（Claude）のアプローチ

Anthropicは「指示・コンテキスト・例・出力形式」の4分割スタイルを推奨しています。

# Anthropic 推奨の4分割スタイル
prompt = """
## 指示（Task）
以下の顧客メールを分析し、緊急度と対応方針を判定してください。

## コンテキスト（Context）
あなたはAxxx Japanのカスタマーサポート担当です。
SLAは「緊急：2時間以内、通常：24時間以内」です。

## 例（Examples）
メール: 「システムが全停止しています。業務ができません」
緊急度: 高 / 対応: 2時間以内にエンジニアと連絡

## 出力形式（Output Format）
JSON形式で: {"urgency": "高|中|低", "action": "対応方針", "deadline": "期限"}

## 分析対象メール
{customer_email}
"""

OpenAI のアプローチ

OpenAI は Tool calling / Structured Outputs の活用でコードとの統合を重視しています。

観点 Anthropic（Claude） OpenAI（GPT）
設計思想 自然言語での詳細な指示を重視 構造的なAPIとツール統合を重視
構造化出力 プロンプトでJSON指示（tool_use） response_format・Pydantic連携
ロール設計 Humanタグ・Assistantタグで明示 system / user / assistant ロール
共通原則 明確・具体的・検証可能なプロンプト設計

共通原則: 明確・具体的・検証可能

  • 明確（Clear）： 一つの指示に複数の意味を持たせない
  • 具体的（Specific）： 「うまくやってください」ではなく数値・形式・条件を明示する
  • 検証可能（Verifiable）： 出力が要件を満たしているか自動テストできる形にする
プロンプトエンジニアリングのベストプラクティスまとめ：
  1. タスク・評価基準・禁止事項の3点セットをシステムプロンプトに含める
  2. 出力フォーマット（JSON等）を明示し、後続コードで確実にパースできる形にする
  3. Few-shotで期待する出力例を2〜3個含める
  4. Pydantic + Structured Outputs で型安全な出力を得る
  5. プロンプトをバージョン管理し、変更前後で品質評価（Evals）を行う
