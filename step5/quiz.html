Step 5 確認テスト — チーム活用とベストプラクティス

所要時間: 約10分  |  問題数: 10問

選択問題

Q1. Claude Code に渡してはいけない情報として誤っているものはどれですか？

正解: c — 匿名化されたサンプルデータは渡しても問題ありません。「誤っているもの」＝「渡してはいけないという記述が誤り」です。API キー・個人情報・顧客データは渡してはいけません。

Q2. 効果的なプロンプト設計として正しいものはどれですか？

正解: b — 「どのファイルの・どの関数に・何の目的で・どんな制約で」を含む具体的なプロンプトが効果的です。「バグを直して」より「src/auth.py の login() で〜」の方が精度が高くなります。

Q3. チームの CLAUDE.md を git リポジトリに含めるメリットはどれですか？

正解: b — CLAUDE.md を git に含めることで、全メンバーがリポジトリをクローンするだけで同じルール・制約・コーディング規約を Claude Code が適用した状態で作業できます。

Q4. 意図しないファイル変更が発生した場合の対処として最も適切なのはどれですか？

正解: b — git で管理していれば git restore . で即座に元に戻せます。再発防止のため CLAUDE.md に「〜は変更しないこと」と明示することが重要です。

Q5. CI/CD での Claude Code 活用として最も安全なアプローチはどれですか？

正解: b — まず影響範囲が小さい読み取り系（コードレビュー・静的解析・テスト実行）から始め、安全性を確認してから徐々に適用範囲を広げるのがベストプラクティスです。

Q6. プロンプトでエラーを伝える場合のベストプラクティスはどれですか？

正解: c — エラーメッセージは全文そのまま貼り付けましょう。自己解釈した要約では重要な情報が抜け落ちます。スタックトレースも含めて貼り付けると診断精度が上がります。

Q7. 「コンテキストが長くなりすぎて応答の質が下がっている」場合の対処はどれですか？

正解: b — /clear で会話履歴をリセットし、現在のタスクに必要な情報だけを再提供します。コンテキストが長すぎると関連性の低い情報がノイズになるためです。

Q8. git 操作で「確認必須」と定めるべき操作として正しい組み合わせはどれですか？

正解: b — git push --force（リモート履歴の破壊）や git reset --hard（ローカル変更の消去）は元に戻しにくい破壊的操作のため、必ず確認が必要です。

Q9. チームへの Claude Code 導入で最初に行うべきステップとして適切なのはどれですか？

正解: b — まず個人利用でベストプラクティスを発見し、チームに合ったガイドライン・CLAUDE.md・settings.json を整備してから展開するのが失敗しない導入手順です。

Q10. AI が生成したコードへの適切な対応はどれですか？

正解: b — AI は誤りを犯すことがあります。生成されたコードは必ず内容を理解し、意図した変更になっているかレビューしてから適用してください。テスト通過だけでは不十分です。
